## 1) 9장) 크리티컬 섹션 보호 도구 4가지

크리티컬 섹션은 여러 스레드가 동시에 접근하면 상태가 망가지는 공유 자원을 다루는 코드 구간이야.  
Pintos/Pintos 스타일에서 자주 쓰는 대표적인 4개 도구를 정리하면 다음이야.

1. **인터럽트 비활성화 (`intr_disable`)**
   - 스레드가 현재 실행 중인 CPU에서 인터럽트를 받지 않게 해서, 타이머 기반 선점(스케줄러 호출)을 잠시 멈춤.
   - 매우 짧은 임계 구간에서만 사용.
2. **락 (`lock`)**
- 한 번에 하나의 스레드만 임계 구간 진입을 허용. 소유자(owner) 개념을 가짐.
3. **세마포어 (`semaphore`)**
   - 카운터 기반 동기화. 자원 개수나 시그널용으로 유용.
4. **조건 변수 (`condition variable`)**
   - 특정 조건이 참이 될 때까지 기다렸다가(`wait`) 신호가 오면 다시 깨어나 처리.
   - 대개 `lock` + `condition variable` 조합으로 사용.

## 2) 인터럽트 disable은 어떻게 선점을 막나? (질문 1)

핵심은 한 문장: **인터럽트가 안 오면 타이머 스케줄러가 스레드를 선점 못한다.**

- 보통 선점은 타이머 인터럽트에서 시작된다.  
- 인터럽트가 꺼져 있으면:
  - 커널이 타임슬라이스 기반으로 현재 스레드를 강제로 내쫓는 동작이 멈춤.
  - 인터럽트 핸들러도 실행되지 않아, 공유 자료구조를 건드리는 중간에 끼어드는 일이 줄어듦.
- 주의:
  - 이것은 `임시로 전역 선점 방지`이지, 영구적 동기화 도구가 아님.
  - 너무 오래 두면 시스템 반응이 나빠지므로 반드시 짧은 구간만 사용.

### 예시
```c
old = intr_disable();
shared_cnt++; // 짧은 임계구간
intr_set_level(old); // 원래 상태 복원
```
- 여기서 `intr_disable`은 “현재 CPU에서 스케줄링 개입을 잠깐 막아” 정확도 있게 공유 데이터를 수정하기 위한 용도야.

## 3) 락과 뮤텍스 차이 (질문: 용어가 2개인지, 언제 쓰는지)

네 질문처럼 “이게 같은 뜻이야?”라고 느낄 수 있어.  
정리하면:

- **같은 성질을 공유한다**: 둘 다 동시에 하나만 접근한다는 점에서는 동일한 **상호배제(mutual exclusion)** 목적을 가진다.
- **중요한 차이**
  - `lock`은 운영체제 커널 내부 동기화 객체 맥락에서 구현/사용이 넓음(구현 방식 자체에 따라 스핀락/뮤텍스/상호배제 객체의 일반화라고 봄).
  - `뮤텍스`는 일반적으로 “오직 **소유자가 lock/unlock 할 수 있는 배타적 접근 도구**”를 강조하는 개념.
  - `세마포어`와의 차이(비교 포인트)는 **카운팅 가능성**과 **의미론**이 달라짐.

### 언제 어떤 걸 쓰나?
- **락**: 임계구간 보호가 필요할 때 기본값.
- **뮤텍스(단일 소유 뮤텍스 의미)**: 스레드가 자원을 독점적으로 잡고 반납해야 하는 패턴(소유권 관리가 명확할 때)에서 쓰기 적합.
- **세마포어**:  
  - 자원이 여러 개일 때(카운팅),  
  - 또는 “특정 작업이 끝난 신호”를 주고받을 때 더 직관적.

### 빠른 비교
- 동작 제약: lock/뮤텍스는 보통 **0/1 허가** + 소유자 개념  
- 세마포어는 **여러 permit**(허가 개수) + 소유자 개념이 본질은 아님  
- 실수 포인트: 단순히 mutual exclusion만 필요할 때 세마포어를 쓰면 의미가 약간 헷갈리기 쉬움.

## 4) 10장) 세마포어 구체 예시

### 예시 A: Producer-Consumer (유한 버퍼)
- `mutex`: 버퍼 접근 보호
- `slots`: 빈 슬롯 개수 (초기값 `N`)
- `items`: 채워진 아이템 개수 (초기값 `0`)

```text
Producer:
  while true:
    wait(slots)      // 빈 공간 기다림
    lock(mutex)
    put_item()
    unlock(mutex)
    signal(items)     // 새 아이템 생김 알림

Consumer:
  while true:
    wait(items)      // 아이템 기다림
    lock(mutex)
    get_item()
    unlock(mutex)
    signal(slots)    // 빈 공간 생김 알림
```
- 이 구조의 포인트: 세마포어가 “자원 수량 관리”와 “누가 깨어날지”를 깔끔히 분리.

### 예시 B: 일회성 배리어(barrier) 느낌
- 여러 스레드가 준비가 끝날 때까지 기다리게 하고, 마지막 스레드가 모두 시작을 허용.
- `ready_count`와 `all_ready`(초기 0)로 구현 가능.

### 핀토스 스타일 코멘트
- `sema_down()`은 permit이 0이면 블록.
- `sema_up()`은 permit 증가 + 대기 스레드 하나 깨움.
- 즉, 이벤트 알림처럼 쓰면 동기화 의도가 명확해짐.

## 5) 12장) “조건 검사와 대기 상태 전환의 원자성”이 뭔가? (질문 12)

문장: “조건 확인(예: `while (조건)`)과 대기(wait) 상태 전환이 원자적이어야 한다.”

의미:  
다른 스레드가 신호를 보내는 순간과, 내가 기다리러 가는 순간 사이에 틈이 있으면 **잃어버린 깨움(lost wakeup)**가 생김.

### 왜 위험한가? (반례)
1) A가 `if (cond == true) ... else sleep()`을 수행
2) A: 조건 검사 직후 **context switch**됨
3) B가 조건을 true로 바꾸고 signal 보냄
4) A가 아직 wait 큐에 안 들어간 상태라 signal을 놓침
5) A는 영원히 잠들어있음

이걸 방지하려면 `조건 검사 + 대기 큐 등록 + 잠들기`를 하나의 흐름(원자성 보장)으로 처리해야 함.

### 핵심 동작 흐름(개념)
- `wait()`은 내부적으로 lock을 반납하고 대기 큐에 등록 후 잠자기로 전환
- signal는 대기 큐에 있는 스레드를 깨우는 쪽과 race가 없게 만들어야 함
- 그래서 대부분:
  - 조건 검사 루프는 `while` 반복
  - 실제 큐 이동은 락/인터럽트 제어 하에서 연속 처리

## 6) 16장) MLFQS 질문 정리

### Q1) 왜 I/O 중심 스레드에 더 빨리 반응하게 하나요?
I/O-bound 스레드는 계산보다 잠자기(대기)가 많아서 최근 CPU 사용량이 낮고,  
MLFQS의 우선순위 공식은 최근 CPU 사용량이 낮은 쪽을 상대적으로 높게 잡아주기 때문에 반응성이 올라간다.

- 직관적으로:
  - CPU만 잘 쓰는 스레드(컴퓨팅 바운드)는 벌점(우선순위 하락)
  - I/O 기다렸다 깨어나는 스레드는 대기 시간이 많아 자주 깨어나는데, 이들이 계속 `recent_cpu` 누적이 적으면 비교적 높은 우선순위를 받음
  - 따라서 인터랙티브(입출력 반응형) 느낌을 개선

### Q2) `nice` 값이 뭔가?
- 사용자가 지정 가능한 “좋음/나쁨” 선호도 값(보통 -20~20).
- 작은 값(음수)일수록 더 많은 CPU를 달라는 의도.
- 공식상 `priority = PRI_MAX - recent_cpu/4 - nice*2` 형태로 반영되어 nice가 크면 우선순위가 내려감.
- 즉, nice는 운영 체제에 대한 **성능 우선순위 힌트**.

### Q3) 시스템 전체 부하를 `load average`로 반영한다는 의미
- `load_avg`는 “최근 시점에서 runnable(실행 준비) + running 스레드가 얼마나 있었는지”의 지수 이동 평균.
- 수치가 크면 시스템이 바쁨 → `load_avg`가 커져 `recent_cpu` 감소 계수에 영향을 줌.
- 한마디로, `load_avg`는 개별 스레드가 아닌 **시스템 혼잡도**를 반영하는 지표야.
- 이 값이 올라가면 개별 스레드의 CPU 사용량이 동일해도 상대 우선순위 재계산에서 더 강하게 감점되어 공평성/안정성이 유지됨.

## 7) 예시 보강 파트 (요청 구간)

### 인터럽트 disable과 락이 충돌할 때는?
- `intr_disable`은 아주 짧은 구간에서만; 긴 대기는 `lock`이나 `semaphore`로 처리.
- 규칙: “짧고 즉시 끝나는 경계 보호는 인터럽트 제어, 일반 임계구간은 락 사용”을 기본으로.

### 락 대신 세마포어를 잘못 쓴 예
- 단순 mutual exclusion에 `semaphore` 1개를 쓸 수는 있으나,  
  다음 코드처럼 상태 변화 + 카운트 의미가 뒤섞이면 오히려 디버깅이 어려워짐.

```text
sema(1) 하나로 모든 보호를 처리 = 공유 데이터 보호도 되고 시그널링도 되는 형태처럼 보이지만
문맥 의도가 혼재되어 실수 포인트 증가
```
- 이때는 `lock` + 필요 시 `cond` 조합이 의도와 가독성이 더 좋다.
