# HY 학습문서 시스템 (Root Index)

이 폴더의 모든 문서는 이 문서를 최우선 소스로 삼습니다.  
새 문서 생성/수정 요청이 오면 **가장 먼저 이 문서를 읽고 규칙 적용**합니다.

## 0) 운영 원칙 (반드시 준수)

1. 루트 최우선 원칙
   - `docs/HY/README.md`를 가장 먼저 읽는다.
   - 루트 문서를 못 읽는 상황이 생기면 먼저 이 문서를 새로 복원/생성하고 기준을 복기한다.
2. 상/하위 링크 의무
   - 모든 학습 문서의 상단에 상위 인덱스 링크와 최소 1개 이상 관련 문서 링크를 둔다.
   - 예시:
     - 상위: `- [상위 인덱스](../README.md)`
     - 관련: `- [OS 스케줄링](./OS/scheduling.md)`
3. 신규 요청 처리 프로토콜(고정)
   - 요청 접수 시 기존 인덱스에서 유사 주제 검색
   - 유사도 낮음: 새 문서 생성 + 인덱스 반영
   - 유사도 높음: 병합/업데이트, 중복 제거 후 정합성 검증
4. 문서 추적성
   - 파일명과 헤더 메타데이터(카테고리/난이도/상태/업데이트일/태그)를 유지한다.
5. 주기적 정렬
   - 인덱스는 학습상태, 난이도, 최근 갱신일 기준으로 정렬 가능하도록 관리한다.

## 1) 1차/2차 읽기 전략(요청하신 핵심 전략)

### 1차 읽기 (BFS: 가볍고 넓게)
- PDF/책을 처음 읽을 때는 모르는 개념을 전부 깊이 파고들지 않는다.
- 챕터 단위로 훑으며 “어느 지점에서 이해가 약한지”를 단순 리스트업한다.
- 이 단계 산출물: 모르는 항목 목록(항목 번호, 위치, 의심 포인트)

### 2차 정리 (요청 기반 병합)
- 요청 받은 목록을 받아,
  1) 원문 위치(챕터/절/페이지/문단 기반)를 확인해 적절한 문단/항목에 배치
  2) 기존 문서에 중복을 피하며 병합
  3) 문서 내 “관련 링크”에 연결
- 핵심은 DFS(한 주제 깊은 잠수) 대신, 각 질문을 점진적으로 쌓아 BFS(진행 기반)로 반영

### 필수 구성

## 3) 유사도 판단 기준(새 문서인지 병합인지)

- 같은 키워드 2개 이상 중첩 + 의미 중복이 뚜렷함: 병합 후보
- 기존 문서에 핵심 개념이 이미 빠짐없이 존재: 병합 업데이트
- 새로운 시점/새 장에서 다른 맥락이거나 반대개념/상보 관계라면 새 문서
- 병합 시: 기존 문단은 축약/정리, 새 내용은 중복 제거 후 보완

## 4) 원문 자료 제공 권장

네가 마지막에 물어본 부분에 대한 답:
- **문서형 학습 자료 원문(텍스트 혹은 핵심 발췌본)을 제공하면 정리 정확도가 높아짐**
- 하지만 원문이 없을 때도, 너의 모르는 항목 목록만 받으면 “구간 기반 보강”은 충분히 수행 가능
- 가능하면 `출처(책/강의명, 챕터, 페이지)`를 같이 주면 문맥 배치 정확도가 높다

## 10) 문서형 학습 리소스 운영 규칙(추가)

- 문서형 리소스는 `docs/HY/resources/` 하위에 보관한다.
- 작업 기본 순서
  - 리소스 등록(`resources/meta` + `toc`)
  - 1차 스캔(모르는 항목 리스트 축적)
  - 2차 요청 기반 정리(원문 위치 기준 병합/삽입)
- 리소스 폴더는 학습 md의 출처 추적 기준이며, 본문 변경 시 반드시 관련 리소스 링크를 남긴다.
- 리소스 상태
  - pending: 등록만 완료
  - studying: 1차 스캔 중
  - reviewed: 2차 반영 후 1차 리뷰 완료
  - archived: 참고만 남김
- 원문 자료는 가능한 한 구조/페이지/절 정보를 함께 받아야 맵핑 정확도가 높다.
- 리소스 반영 규칙
  - 모르는 항목은 즉시 1차 로그만 기록(과도한 즉시 정리 지양)
  - 2차 요청 시에만 학습 md 본문에 깊이 반영

## 8) 시각 자료 운영 규칙

### 사용 가능한 형식
- Mermaid (권장): 흐름/의존성/상태/타임라인 정리에 최적
- SVG: 정밀한 다이어그램, 아이콘, 구조도, 비교표 등을 직접 임베드할 때 사용
- 수식(LaTeX/KaTeX): 수학식, 점화식, 복잡한 식 표현
- 표(테이블): 동작 비교, 속성 비교, API/파라미터 정리에 적합
- 체크리스트/타임라인: 학습 체크리스트, 복습 주기, 실험 절차 관리
- 최소 이미지(로컬 PNG/SVG): 교재의 흐름 캡처, UI 캡처, 실행 결과 화면

### 사용 가이드(오버헤드 최소화)
- 개념 간 상호작용이 3개 이상이면 다이어그램 우선
- 분기/옵션이 많은 절차는 Mermaid flowchart 사용
- 개념 간 상태 전이를 다룰 때는 Mermaid state/sequence 다이어그램 사용
- 표로 5개 이하는 비교 단락, 6개 이상은 시각화(표+한두 개 키포인트)로 분리
- 꼭 필요한 경우에만 외부 이미지 사용(저작권·출처·캡션 명시 필수)
- 시각 자료는 본문 뒤미에 몰아넣지 말고 핵심 개념 단락 바로 아래에 배치

### 문서 내 작성 규칙
- 모든 시각 자료 블록에 캡션(한 줄 설명)과 소스 위치(챕터/절/페이지 or 참고자료)를 남긴다.
- Mermaid는 코드블록으로 직접 작성한다. 가능하면 렌더링 가능한 문서 포맷 기준(예: GitHub, VSCode, Obsidian/Typora 호환성)을 고려한다.
- SVG는 `assets/` 폴더에 저장하고 문서에서는 상대경로 또는 절대경로로 링크한다.
- 동일 개념에서 Mermaid와 SVG를 동시에 쓰는 것은 피하고, 하나의 핵심 표현으로 통일한다.

### 권장 디렉터리 구조 예시
- `docs/HY/assets/svg/` : 외부에서 저장한 SVG/이미지
- `docs/HY/assets/diagrams/` : Mermaid 추출본 또는 스크린샷
- `docs/HY/assets/notes/` : 표지, 캡처, 실습 결과 이미지
- 문서 하단에 자료별 출처/라이선스 메모 추가

### 추가 제안
- 단편 암기용 개념은 `카드형 정리 블록`(요약 + 반례 + 한 줄 키워드)로 정리하면 재회독 효율이 높다.
- 장(챕터) 단위 학습엔 `문제-해결-요약` 3단계 구조를 추천한다.

## 9) 예시 중심 설명 원칙

- 추상 정의만 나열하지 말고, 개념이 어렵거나 오개념이 생기기 쉬운 구간에서는 반드시 **구체 사례**를 함께 제시한다.
- 기본 원칙
  - `필수 예시`: 새 개념/새 용어, 반대 개념 대비, 조건/예외가 있는 규칙
  - `선택 예시`: 일반적인 개념 반복, 직관적 개념
  - `요청형 예시`: 사용자가 “이 부분 어렵다/예시 줘” 요청 시 즉시 해당 구간 보강
- 예시 작성 규칙
  - 예시 제목은 “예시”로 통일하고, 가능하면 짧은 입력/출력 또는 상황-결과-왜 그런지 구조로 작성
  - 오개념이 생기기 쉬운 포인트는 `반례`를 한 줄 덧붙여 명확화
- 반영 방식
  - 새 문서 생성 시 1~2개 정도의 핵심 예시를 우선 배치
  - 기존 문서 업데이트 시 요청이 오면, 해당 문단/소제목 바로 아래에 `예시` 파트를 삽입
- 사용자 요청 규약
  - “이해가 안 갔던 부분”, “예시 더”처럼 모호한 요청도 즉시 `해당 구간 집중 예시`로 해석해 대응
