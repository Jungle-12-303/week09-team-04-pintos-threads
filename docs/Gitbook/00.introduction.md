# 프로젝트 2: 사용자 프로그램

이제 Pintos를 다루어 보았고 Pintos의 인프라와 스레드 패키지에 익숙해지고 있으므로, 사용자 프로그램을 실행할 수 있게 하는 시스템 부분을 작업할 차례입니다. 기본 코드는 이미 사용자 프로그램을 로드하고 실행하는 기능을 지원하지만, I/O나 상호작용은 불가능합니다. 이 프로젝트에서는 프로그램이 시스템 호출을 통해 OS와 상호작용할 수 있도록 만들 것입니다. 이번 과제에서는 주로 `userprog` 디렉터리에서 작업하게 되지만, Pintos의 거의 모든 다른 부분과도 상호작용하게 됩니다. 관련된 부분은 아래에서 설명합니다.

프로젝트 2는 프로젝트 1 제출물 위에 구축해야 합니다. 프로젝트 1의 코드가 프로젝트 2 코드에 영향을 주지는 않지만, 이 프로젝트는 누적형 프로젝트이므로 프로젝트 1 테스트 케이스도 계속 통과해야 합니다.

또한 프로젝트 2에는 추가 도전 과제가 있다는 점에 유의하세요. 이는 선택 구현 사항입니다. 추가 도전 과제에는 테스트 케이스를 제외하면 스켈레톤 코드가 없습니다. 모든 설계는 여러분에게 달려 있습니다. 추가 요구사항을 제출하고 테스트하려면 `userprog/Make.vars`를 수정해야 합니다.

마지막으로, `TODO`가 없는 코드라고 해서 항상 수정할 필요가 없다는 뜻은 아닙니다. 테스트 코드를 제외하고는 프로젝트 2의 어떤 소스 코드든 자유롭게 수정할 수 있습니다.

## 배경

지금까지 Pintos에서 실행한 모든 코드는 운영체제 커널의 일부였습니다. 예를 들어, 이전 과제의 모든 테스트 코드는 커널의 일부로 실행되었고, 시스템의 특권 영역에 완전하게 접근할 수 있었습니다. 운영체제 위에서 사용자 프로그램을 실행하기 시작하면 더 이상 그렇지 않습니다. 이 프로젝트는 그 결과로 생기는 문제들을 다룹니다.

우리는 한 번에 둘 이상의 프로세스가 실행되는 것을 허용합니다. 각 프로세스는 하나의 스레드를 가집니다. 멀티스레드 프로세스는 지원하지 않습니다. 사용자 프로그램은 자신이 전체 머신을 가진 것처럼 착각하도록 작성됩니다. 즉, 여러 프로세스를 동시에 로드하고 실행할 때 이 착각을 유지하려면 메모리, 스케줄링, 그리고 다른 상태를 올바르게 관리해야 합니다.

이전 프로젝트에서는 테스트 코드를 커널에 직접 컴파일했기 때문에, 커널 내부의 특정 함수 인터페이스를 요구해야 했습니다. 이제부터는 사용자 프로그램을 실행하여 운영체제를 테스트합니다. 이 방식은 훨씬 더 큰 자유를 제공합니다. 사용자 프로그램 인터페이스가 여기에서 설명하는 명세를 만족하도록 해야 하지만, 그 제약 안에서는 원하는 대로 커널 코드를 재구성하거나 다시 작성할 수 있습니다.

여러분의 코드는 절대로 `#ifdef VM`으로 둘러싸인 블록 안에 위치해서는 안 됩니다. 이 블록들은 프로젝트 3에서 구현할 가상 메모리 하위 시스템을 활성화한 뒤 포함됩니다. 또한 코드가 `#ifndef VM`으로 둘러싸여 있으면 프로젝트 3에서는 그 코드가 제외됩니다.

시작하기 전에 동기화와 가상 주소에 대해 읽어 보는 것을 강력히 권장합니다.

## 소스 파일

여러분이 하게 될 프로그래밍 작업의 개요를 파악하는 가장 쉬운 방법은 작업하게 될 각 부분을 살펴보는 것입니다. `userprog`에는 적은 수의 파일만 있지만, 대부분의 작업은 여기에서 이루어집니다.

### `process.c`, `process.h`

ELF 바이너리를 로드하고 프로세스를 시작합니다.

### `syscall.c`, `syscall.h`

사용자 프로세스가 커널 기능에 접근하려 할 때마다 시스템 호출을 호출합니다. 이 파일들은 스켈레톤 시스템 호출 핸들러입니다. 현재는 메시지를 출력하고 사용자 프로세스를 종료하기만 합니다. 이 프로젝트의 파트 2에서는 시스템 호출에 필요한 나머지 모든 작업을 수행하는 코드를 추가하게 됩니다.

### `syscall-entry.S`

시스템 호출 핸들러를 부트스트랩하는 작은 어셈블리 코드입니다. 이 코드를 이해할 필요는 없습니다.

### `exception.c`, `exception.h`

사용자 프로세스가 특권 명령이나 금지된 동작을 수행하면 예외 또는 fault로 커널에 트랩됩니다. 이 파일들은 예외를 처리합니다. 현재 모든 예외는 단순히 메시지를 출력하고 프로세스를 종료합니다. 프로젝트 2의 일부 해법은 이 파일의 `page_fault()`를 수정해야 하지만, 모든 해법이 그런 것은 아닙니다.

### `gdt.c`, `gdt.h`

x86-64는 세그먼트 구조를 가진 아키텍처입니다. Global Descriptor Table, 즉 GDT는 사용 중인 세그먼트를 설명하는 테이블입니다. 이 파일들은 GDT를 설정합니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없습니다. GDT가 어떻게 동작하는지 궁금하다면 코드를 읽어 볼 수 있습니다.

### `tss.c`, `tss.h`

Task-State Segment, 즉 TSS는 x86 아키텍처의 태스크 전환에 사용되었습니다. 그러나 x86-64에서는 태스크 전환이 더 이상 권장되지 않습니다. 그럼에도 TSS는 여전히 링 전환 중 스택 포인터를 찾기 위해 존재합니다.

이는 사용자 프로세스가 인터럽트 핸들러에 진입할 때, 하드웨어가 커널의 스택 포인터를 찾기 위해 TSS를 참조한다는 뜻입니다. 어떤 프로젝트에서도 이 파일들을 수정할 필요는 없습니다. TSS가 어떻게 동작하는지 궁금하다면 코드를 읽어 볼 수 있습니다.

## 파일 시스템 사용

이 프로젝트에서는 파일 시스템 코드와 상호작용해야 합니다. 사용자 프로그램은 파일 시스템에서 로드되며, 구현해야 하는 많은 시스템 호출이 파일 시스템을 다루기 때문입니다. 그러나 이 프로젝트의 초점은 파일 시스템이 아니므로, `filesys` 디렉터리에 단순하지만 완전한 파일 시스템을 제공했습니다. 파일 시스템 사용 방법, 특히 여러 제한 사항을 이해하려면 `filesys.h`와 `file.h` 인터페이스를 살펴보는 것이 좋습니다.

이 프로젝트에서는 파일 시스템 코드를 수정할 필요가 없으며, 수정하지 않는 것을 권장합니다. 파일 시스템 작업은 이 프로젝트의 초점에서 벗어나게 만들 가능성이 큽니다.

지금 파일 시스템 루틴을 올바르게 사용하면 프로젝트 4에서 파일 시스템 구현을 개선할 때 훨씬 수월해집니다. 그때까지는 다음 제한 사항을 감수해야 합니다.

- 내부 동기화가 없습니다. 동시 접근은 서로 간섭합니다. 한 번에 하나의 프로세스만 파일 시스템 코드를 실행하도록 동기화를 사용해야 합니다.
- 파일 크기는 생성 시점에 고정됩니다. 루트 디렉터리는 파일로 표현되므로 생성할 수 있는 파일 수도 제한됩니다.
- 파일 데이터는 하나의 extent로 할당됩니다. 즉, 하나의 파일에 있는 데이터는 디스크의 연속된 섹터 범위를 차지해야 합니다. 따라서 파일 시스템을 오래 사용할수록 외부 단편화가 심각한 문제가 될 수 있습니다.
- 서브디렉터리가 없습니다.
- 파일 이름은 14자로 제한됩니다.
- 작업 도중 시스템이 크래시되면 디스크가 자동 복구 불가능한 방식으로 손상될 수 있습니다. 어차피 파일 시스템 복구 도구도 없습니다.

중요한 기능 하나는 포함되어 있습니다.

- `filesys_remove()`에는 Unix와 유사한 의미론이 구현되어 있습니다. 즉, 파일이 열려 있는 상태에서 제거되면 그 블록들은 즉시 해제되지 않으며, 마지막으로 열고 있던 스레드가 닫을 때까지 해당 파일을 열고 있는 모든 스레드가 계속 접근할 수 있습니다. 자세한 내용은 "Removing an Open File"을 참고하세요.

프로젝트 1에서는 모든 테스트 프로그램이 이미 커널 이미지 안에 있었지만, 이번에는 사용자 공간에서 실행되는 테스트 프로그램을 Pintos 가상 머신 안에 넣어야 합니다. 작업을 훨씬 쉽게 하기 위해 `make check` 같은 테스트 스크립트가 이를 자동으로 처리하므로, 대부분의 경우 이 과정을 이해할 필요는 없습니다. 하지만 이를 알고 있으면 개별 테스트 케이스를 실행하는 데 큰 도움이 됩니다.

Pintos 가상 머신에 파일을 넣으려면 먼저 파일 시스템 파티션이 있는 시뮬레이션 디스크를 만들 수 있어야 합니다. `pintos-mkdisk` 프로그램이 이 기능을 제공합니다. `userprog/build` 디렉터리에서 다음을 실행합니다.

```sh
pintos-mkdisk filesys.dsk 2
```

이 명령은 2 MB Pintos 파일 시스템 파티션을 포함하는 `filesys.dsk`라는 시뮬레이션 디스크를 만듭니다. 그런 다음 `pintos` 뒤, `--` 앞에 `--fs-disk filesys.dsk`를 전달하여 디스크를 지정합니다. 예시는 다음과 같습니다.

```sh
pintos --fs-disk filesys.dsk -- KERNEL_COMMANDS...
```

`--`가 필요한 이유는 `--fs-disk`가 시뮬레이션 커널이 아니라 `pintos` 스크립트를 위한 옵션이기 때문입니다. 그다음 커널 명령줄에 `-f -q`를 전달하여 파일 시스템 파티션을 포맷합니다.

```sh
pintos SCRIPT_COMMANDS -- -f -q
```

`-f` 옵션은 파일 시스템을 포맷하게 하고, `-q`는 포맷이 끝나자마자 Pintos가 종료되도록 합니다.

시뮬레이션 파일 시스템 안팎으로 파일을 복사할 방법도 필요합니다. `pintos -p`("put")와 `-g`("get") 옵션이 이 작업을 수행합니다. `file`을 Pintos 파일 시스템으로 복사하려면 다음을 사용합니다.

```sh
pintos -p file -- -q
```

Pintos 파일 시스템에 `newname`이라는 이름으로 복사하려면 원래 파일 이름 뒤에 `:newname`을 추가합니다.

```sh
pintos -p file:newname -- -q
```

VM 밖으로 파일을 복사하는 명령도 비슷하지만, `-p` 대신 `-g`를 사용합니다.

참고로 이 명령들은 커널 명령줄에 `extract`와 `append`라는 특별한 명령을 전달하고, 특별한 시뮬레이션 `scratch` 파티션과 파일을 주고받는 방식으로 동작합니다. 구현 세부 사항이 매우 궁금하다면 `pintos` 스크립트와 `filesys/fsutil.c`를 살펴볼 수 있습니다.

다음은 파일 시스템 파티션이 있는 디스크를 만들고, 파일 시스템을 포맷하고, 이 프로젝트의 두 번째 테스트 케이스인 `args-single` 프로그램을 새 디스크에 복사한 뒤, 인자 `onearg`를 전달하여 실행하는 방법의 요약입니다. 인자 전달은 여러분이 구현하기 전까지는 동작하지 않습니다. 이미 테스트 케이스를 빌드했고 현재 디렉터리가 `userprog/build`라고 가정합니다.

```sh
pintos-mkdisk filesys.dsk 10
pintos --fs-disk filesys.dsk -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

나중에 사용하거나 검사하기 위해 파일 시스템 디스크를 보관하고 싶지 않다면, 네 단계를 하나의 명령으로 합칠 수도 있습니다. `--filesys-size=n` 옵션은 Pintos 실행 동안에만 약 `n` MB 크기의 임시 파일 시스템 파티션을 만듭니다. Pintos 자동 테스트 스위트는 이 문법을 광범위하게 사용합니다.

```sh
pintos --fs-disk=10 -p tests/userprog/args-single:args-single -- -q -f run 'args-single onearg'
```

## 사용자 프로그램의 동작 방식

Pintos는 메모리에 들어가고 여러분이 구현한 시스템 호출만 사용하는 한 일반 C 프로그램을 실행할 수 있습니다. 특히 `malloc()`은 구현할 수 없습니다. 이 프로젝트에서 요구하는 시스템 호출 중 메모리 할당을 허용하는 것이 없기 때문입니다. 또한 Pintos는 부동소수점 연산을 사용하는 프로그램을 실행할 수 없습니다. 스레드를 전환할 때 커널이 프로세서의 부동소수점 장치를 저장하고 복원하지 않기 때문입니다.

Pintos는 `userprog/process.c`에 제공된 로더를 사용하여 ELF 실행 파일을 로드할 수 있습니다. ELF는 Linux, Solaris, 그리고 많은 다른 운영체제에서 오브젝트 파일, 공유 라이브러리, 실행 파일에 사용하는 파일 형식입니다.

실제로는 x86-64 ELF 실행 파일을 출력하는 어떤 컴파일러와 링커든 사용하여 Pintos용 프로그램을 만들 수 있습니다. 우리가 제공한 컴파일러와 링커도 충분히 잘 동작할 것입니다. 테스트 프로그램을 시뮬레이션 파일 시스템에 복사하기 전까지 Pintos는 유용한 일을 할 수 없다는 점을 바로 알아야 합니다. 다양한 프로그램을 파일 시스템에 복사하기 전까지는 흥미로운 작업을 할 수 없습니다. 디버깅 중에는 가끔 `filesys.dsk`가 더 이상 쓸모없는 상태로 망가질 수 있으므로, 깨끗한 기준 파일 시스템 디스크를 만들어 두고 필요할 때 복사해서 사용하는 것이 좋습니다.

## 가상 메모리 배치

Pintos의 가상 메모리는 사용자 가상 메모리와 커널 가상 메모리, 두 영역으로 나뉩니다. 사용자 가상 메모리는 가상 주소 0부터 `KERN_BASE`까지의 범위입니다. `KERN_BASE`는 `include/threads/vaddr.h`에 정의되어 있으며 기본값은 `0x8004000000`입니다. 커널 가상 메모리는 나머지 가상 주소 공간을 차지합니다.

사용자 가상 메모리는 프로세스마다 따로 존재합니다. 커널이 한 프로세스에서 다른 프로세스로 전환할 때, 프로세서의 페이지 디렉터리 기준 레지스터를 변경하여 사용자 가상 주소 공간도 함께 전환합니다. `thread/mmu.c`의 `pml4_activate()`를 참고하세요. `struct thread`는 프로세스의 페이지 테이블을 가리키는 포인터를 포함합니다.

커널 가상 메모리는 전역입니다. 어떤 사용자 프로세스나 커널 스레드가 실행 중인지와 관계없이 항상 같은 방식으로 매핑됩니다. Pintos에서 커널 가상 메모리는 `KERN_BASE`부터 시작하여 물리 메모리에 일대일로 매핑됩니다. 즉, 가상 주소 `KERN_BASE`는 물리 주소 0에 접근하고, 가상 주소 `KERN_BASE + 0x1234`는 물리 주소 `0x1234`에 접근하며, 이런 방식으로 머신의 물리 메모리 크기까지 이어집니다.

사용자 프로그램은 자기 자신의 사용자 가상 메모리에만 접근할 수 있습니다. 커널 가상 메모리에 접근하려고 하면 `userprog/exception.c`의 `page_fault()`가 처리하는 page fault가 발생하고, 해당 프로세스는 종료됩니다. 커널 스레드는 커널 가상 메모리와, 사용자 프로세스가 실행 중이라면 그 실행 중인 프로세스의 사용자 가상 메모리 모두에 접근할 수 있습니다. 그러나 커널 안에서도 매핑되지 않은 사용자 가상 주소의 메모리에 접근하려고 하면 page fault가 발생합니다.

## 일반적인 메모리 배치

개념적으로 각 프로세스는 자신의 사용자 가상 메모리를 원하는 방식으로 자유롭게 배치할 수 있습니다. 실제로 사용자 가상 메모리는 다음과 같이 배치됩니다.

```text
USER_STACK +----------------------------------+
           |            사용자 스택           |
           |                 |                |
           |                 |                |
           |                 V                |
           |            아래로 성장           |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |            위로 성장             |
           |                 ^                |
           |                 |                |
           |                 |                |
           +----------------------------------+
           | 초기화되지 않은 데이터 세그먼트 |
           +----------------------------------+
           |      초기화된 데이터 세그먼트    |
           +----------------------------------+
           |            코드 세그먼트         |
 0x400000  +----------------------------------+
           |                                  |
           |                                  |
           |                                  |
           |                                  |
           |                                  |
       0   +----------------------------------+
```

이 프로젝트에서 사용자 스택의 크기는 고정되어 있지만, 프로젝트 3에서는 스택이 성장할 수 있게 됩니다. 전통적으로는 시스템 호출로 초기화되지 않은 데이터 세그먼트의 크기를 조정할 수 있지만, 여러분은 이를 구현할 필요가 없습니다.

Pintos의 코드 세그먼트는 사용자 가상 주소 `0x400000`에서 시작합니다. 이는 주소 공간의 바닥에서 약 128 MB 떨어진 위치입니다. 이 값은 Ubuntu에서 일반적이었던 값이며, 깊은 의미는 없습니다.

링커는 프로그램 세그먼트들의 이름과 위치를 알려 주는 링커 스크립트의 지시에 따라 사용자 프로그램의 메모리 배치를 설정합니다. 링커 스크립트에 대해 더 알고 싶다면 `info ld`를 통해 접근할 수 있는 링커 매뉴얼의 "Scripts" 장을 읽어 보세요.

특정 실행 파일의 배치를 보려면 `objdump`를 `-p` 옵션과 함께 실행하세요.

## 사용자 메모리 접근

시스템 호출의 일부로, 커널은 사용자 프로그램이 제공한 포인터를 통해 메모리에 접근해야 하는 경우가 많습니다. 이때 커널은 매우 조심해야 합니다. 사용자가 null 포인터, 매핑되지 않은 가상 메모리를 가리키는 포인터, 또는 `KERN_BASE` 위의 커널 가상 주소 공간을 가리키는 포인터를 전달할 수 있기 때문입니다. 이러한 모든 잘못된 포인터는 커널이나 다른 실행 중인 프로세스에 피해를 주지 않고 거부되어야 하며, 이를 위해 문제가 되는 프로세스를 종료하고 그 자원을 해제해야 합니다.

이를 올바르게 처리하는 합리적인 방법은 적어도 두 가지가 있습니다.

첫 번째 방법은 사용자가 제공한 포인터의 유효성을 검증한 다음 역참조하는 것입니다. 이 방법을 선택한다면 `thread/mmu.c`와 `include/threads/vaddr.h`에 있는 함수들을 살펴보는 것이 좋습니다. 이는 사용자 메모리 접근을 처리하는 가장 단순한 방법입니다.

두 번째 방법은 사용자 포인터가 `KERN_BASE` 아래를 가리키는지만 확인한 다음 역참조하는 것입니다. 잘못된 사용자 포인터는 page fault를 일으키며, 이는 `userprog/exception.c`의 `page_fault()` 코드를 수정하여 처리할 수 있습니다. 이 기법은 프로세서의 MMU를 활용하므로 보통 더 빠르며, Linux를 포함한 실제 커널에서 자주 사용됩니다.

어느 방법을 사용하든 자원이 누수되지 않도록 해야 합니다. 예를 들어 시스템 호출이 lock을 획득했거나 `malloc()`으로 메모리를 할당했다고 가정해 봅시다. 그 뒤 잘못된 사용자 포인터를 발견하더라도 lock을 해제하거나 메모리 페이지를 해제해야 합니다. 역참조하기 전에 사용자 포인터를 검증하는 방법을 선택했다면 이는 비교적 간단합니다. 잘못된 포인터가 page fault를 일으키는 경우에는 메모리 접근에서 오류 코드를 반환할 방법이 없으므로 처리하기가 더 어렵습니다. 따라서 후자의 기법을 시도하려는 사람들을 위해 약간의 도움 코드를 제공합니다.

```c
/* 사용자 가상 주소 UADDR에서 1바이트를 읽는다.
 * UADDR는 반드시 KERN_BASE보다 아래에 있어야 한다.
 * 성공하면 바이트 값을 반환하고, segfault가 발생하면 -1을 반환한다. */
static int64_t
get_user (const uint8_t *uaddr) {
    int64_t result;
    __asm __volatile (
    "movabsq $done_get, %0\n"
    "movzbq %1, %0\n"
    "done_get:\n"
    : "=&a" (result) : "m" (*uaddr));
    return result;
}

/* 사용자 주소 UDST에 BYTE를 쓴다.
 * UDST는 반드시 KERN_BASE보다 아래에 있어야 한다.
 * 성공하면 true를 반환하고, segfault가 발생하면 false를 반환한다. */
static bool
put_user (uint8_t *udst, uint8_t byte) {
    int64_t error_code;
    __asm __volatile (
    "movabsq $done_put, %0\n"
    "movb %b2, %1\n"
    "done_put:\n"
    : "=&a" (error_code), "=m" (*udst) : "q" (byte));
    return error_code != -1;
}
```

이 함수들은 각각 사용자 주소가 이미 `KERN_BASE`보다 아래에 있음이 검증되었다고 가정합니다. 또한 커널에서 page fault가 발생했을 때 단순히 `rax`를 `-1`로 설정하고 이전 값을 `%rip`에 복사하도록 `page_fault()`를 수정했다고 가정합니다.
